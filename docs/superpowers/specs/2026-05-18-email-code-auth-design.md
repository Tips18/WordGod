# 邮箱验证码认证设计

## 背景

当前 V1 PRD 将“邮件验证码与找回密码”列为不做范围，但本轮需求确认要同时支持注册邮箱验证、邮箱验证码登录、找回与重置密码。因此本设计将邮件验证码升级为认证模块的统一能力，并要求同步调整 PRD、架构、数据库和 README。

现有认证模型是邮箱密码注册/登录、HttpOnly access cookie、refresh session cookie，以及 `AuthSession` 持久化刷新会话。新能力应复用这个会话模型，不改变阅读检测、生词本、题库导入等业务边界。

## 目标

- 注册前必须先向邮箱发送验证码，注册提交时校验验证码、邮箱和密码。
- 登录页保留邮箱密码登录，同时新增邮箱验证码登录。
- 找回密码通过邮箱验证码确认账号归属，再设置新密码。
- 所有验证码共用一套生成、发送、校验、过期、尝试次数和作废逻辑。
- 邮件发送可以在生产使用 SMTP，在本地开发和测试中使用 console/mock sender。
- 内存存储和 Prisma/PostgreSQL 存储行为一致。

## 非目标

- 不新增短信验证码、第三方 OAuth、支付或商业化能力。
- 不把验证码登录改造成强制二次验证；密码登录仍可直接登录。
- 不引入正式 CMS 或后台管理页面。
- 不新增多考试题库前台筛选。

## 推荐方案

采用统一验证码服务和 SMTP 适配器。

后端新增 `EmailCodeService` 负责验证码生命周期，新增 `EmailSender` 抽象隔离邮件发送实现。业务服务通过 `EmailCodeService` 发送和验证验证码，验证成功后继续使用现有 `AuthService` 会话签发逻辑。这样可以让注册、验证码登录、重置密码共享防滥用规则，也可以用测试替身覆盖邮件发送。

## 用户流程

### 注册邮箱验证

1. 用户在注册模式输入邮箱。
2. 点击“发送验证码”。
3. 后端校验邮箱格式、确认该邮箱尚未注册，生成 6 位数字验证码并发送邮件。
4. 用户输入验证码和密码后提交注册。
5. 后端校验验证码、邮箱和密码，通过后创建用户并签发 access/refresh cookies。
6. 顶部导航立即切换为“退出登录”。

### 邮箱验证码登录

1. 用户在登录页切换到“验证码登录”。
2. 输入已注册邮箱并发送验证码。
3. 后端确认邮箱已注册，生成并发送登录验证码。
4. 用户输入验证码后提交登录，可选择“30天内记住登录”。
5. 后端校验验证码后签发现有会话 cookies。

### 找回与重置密码

1. 用户点击“忘记密码”。
2. 输入已注册邮箱并发送重置密码验证码。
3. 用户输入验证码和新密码。
4. 后端校验验证码后更新 `User.passwordHash`。
5. 重置成功后可选择直接登录并签发 cookies；第一版建议直接登录，减少用户来回跳转。

## API 设计

### 发送验证码

`POST /auth/email-codes`

请求：

```json
{
  "email": "reader@example.com",
  "purpose": "register"
}
```

`purpose` 可选值：`register`、`login`、`reset-password`。

响应：

```json
{
  "success": true
}
```

发送接口对外统一返回成功文案，避免暴露账号是否存在；服务端内部仍按用途校验邮箱状态，并通过日志和测试覆盖分支。若邮箱格式不合法或发送频率过高，返回明确错误。

### 注册

`POST /auth/register`

新增字段：

```json
{
  "email": "reader@example.com",
  "password": "Passw0rd!",
  "emailCode": "123456"
}
```

注册成功响应沿用 `{ "user": { "id": "...", "email": "..." } }`，并写入现有认证 cookies。

### 验证码登录

`POST /auth/login/email-code`

请求：

```json
{
  "email": "reader@example.com",
  "emailCode": "123456",
  "rememberLogin": true
}
```

响应沿用 `AuthResponse`，会话有效期规则与密码登录一致。

### 重置密码

`POST /auth/password/reset`

请求：

```json
{
  "email": "reader@example.com",
  "emailCode": "123456",
  "newPassword": "NewPassw0rd!"
}
```

响应沿用 `AuthResponse` 并写入现有认证 cookies。

## 验证码规则

- 验证码为 6 位数字。
- 存储 `codeHash`，不存明文。
- 默认有效期 10 分钟。
- 同邮箱同用途 60 秒内禁止重复发送。
- 单条验证码最多校验 5 次。
- 校验成功后立即标记为已消费。
- 新发送同邮箱同用途验证码时，上一条未消费验证码作废。
- 所有邮箱入参沿用现有 `trim().toLowerCase()` 规整和格式校验。
- 密码仍要求非空；若未来要增强密码复杂度，应作为独立需求处理。

## 数据库设计

新增 `EmailVerificationCode`：

- `id`
- `email`
- `purpose`
- `codeHash`
- `expiresAt`
- `consumedAt`
- `attemptCount`
- `lastSentAt`
- `createdAt`

索引建议：

- `email + purpose + createdAt`，用于查找最新验证码。
- `expiresAt`，用于未来清理过期验证码。

内存存储新增同等记录集合，Prisma 存储新增对应模型和迁移。该表不关联 `User` 外键，因为注册验证码发生在用户创建前。

## 邮件发送配置

生产环境使用 SMTP：

- `SMTP_HOST`
- `SMTP_PORT`
- `SMTP_USER`
- `SMTP_PASS`
- `SMTP_FROM`
- `SMTP_SECURE`

本地开发默认使用 console sender：不真实发邮件，只在后端日志输出收件邮箱、用途和验证码。测试使用 mock sender 捕获邮件内容，不依赖外部网络。

第一版邮件正文保持简单：

- 标题：`我不是词神验证码`
- 正文包含验证码、用途说明、10 分钟有效期提示。

## 前端设计

`AuthFormCard` 从单一“登录/注册”表单扩展为三个认证面板：

- 密码登录：邮箱、密码、30 天记住登录。
- 验证码登录：邮箱、验证码、发送验证码、30 天记住登录。
- 注册：邮箱、验证码、发送验证码、密码。
- 找回密码：邮箱、验证码、发送验证码、新密码。

按钮状态：

- 发送验证码后进入 60 秒倒计时。
- 发送中和提交中分别禁用相关按钮。
- 后端错误使用现有错误提示区显示。

视觉保持当前纸张面板、暖色状态标签和主按钮风格，不新增营销式页面结构。

## 错误处理

- 邮箱格式不合法：提示“请输入有效邮箱”。
- 发送过于频繁：提示“验证码发送太频繁，请稍后再试”。
- 验证码错误或过期：提示“验证码错误或已过期”。
- 验证次数过多：提示“验证码尝试次数过多，请重新获取”。
- 邮件服务不可用：提示“验证码发送失败，请稍后重试”。

登录和重置密码发送接口不向前端区分邮箱是否已注册，以减少账号枚举风险。提交阶段可返回统一的验证码错误或凭据错误。

## 测试策略

后端单测：

- 发送验证码会生成 6 位数字并调用 sender。
- 同邮箱同用途 60 秒内不能重复发送。
- 验证码过期、错误、次数过多、已消费均失败。
- 注册必须带有效 register 验证码。
- 验证码登录成功签发与密码登录一致的会话时长。
- 重置密码成功后旧密码失效，新密码可登录。

后端 e2e：

- `POST /auth/email-codes`、`POST /auth/register`、`POST /auth/login/email-code`、`POST /auth/password/reset` 主流程。

前端行为测试：

- 注册发送验证码并提交。
- 验证码登录发送验证码并保留 rememberLogin。
- 忘记密码重置成功后导航进入已登录状态。
- 发送验证码按钮倒计时和错误提示。

## 风险与缓解

- 邮件发送供应商配置错误会影响用户注册和找回密码：本地使用 console sender，生产启动文档明确环境变量。
- 验证码可能被暴力尝试：限制尝试次数、有效期和重新发送频率。
- 账号枚举风险：发送接口统一成功响应，提交接口尽量使用统一错误文案。
- 工作区已有大量未提交改动：实现时按认证模块和文档分批提交，避免混入内容管线或题库改动。
